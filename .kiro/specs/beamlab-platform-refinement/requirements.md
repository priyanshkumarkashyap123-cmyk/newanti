# Requirements Document

## Introduction

BeamLab Ultimate is a professional structural engineering SaaS platform. This document is the
single consolidated refinement spec covering the full platform end-to-end. It audits what has
been implemented across all prior specs, identifies remaining gaps, and defines requirements
needed to deliver a polished, production-ready product.

The scope spans seven areas: UI/UX polish, house plan generator accuracy (user-reported
inaccuracy), GPO/high-compute analysis pipeline, report generation, data management, space
planning tools, and modeling tools.

## Glossary

- **Platform**: The full BeamLab Ultimate product including all frontend pages, all three
  backend services (Node_API, Rust_API, Python_API), and the WASM solver.
- **Node_API**: Express 5 Node.js backend (port 3001) — auth, billing, projects, quota.
- **Rust_API**: Axum-based Rust backend (port 8080) — medium-to-large structural analysis.
- **Python_API**: FastAPI-based Python backend (port 8000) — design checks and job queuing.
- **WASM_Solver**: In-browser WebAssembly finite-element solver for small static models.
- **Analysis_Router**: The unified `useAnalysis` hook routing jobs to the correct backend.
- **SpacePlanningEngine**: Client-side TypeScript class generating floor plans, grids, MEP.
- **ArchitecturalPlacement**: The placement pipeline (buildCirculationSpine → placePublicZone
  → placeWetAreas → placePrivateZone) from the realistic-house-plan-generator spec.
- **NBC_MIN_DIMS**: Lookup table mapping RoomType to minimum clear internal dimensions per NBC
  India 2016.
- **PlacedRoom**: A room with resolved x, y, width, height coordinates in metres.
- **Buildable_Envelope**: Inner rectangle after subtracting all setbacks from the plot.
- **Tier_Config**: Configuration object defining feature flags and quota limits per tier.
- **TierGate**: React component wrapping gated UI elements; shows UpgradeModal when blocked.
- **UpgradeModal**: Modal dialog presenting upgrade prompts with feature name and CTA.
- **Subscription_Provider**: React context provider exposing the current user's tier.
- **PDF_Report**: Downloadable PDF generated by PDFReportService or ProfessionalReportGenerator.
- **Quality_Check**: Pass/fail assertion derived from actual analysis and design results.
- **Quota_Service**: Backend service tracking per-user daily usage limits.
- **Collaboration_Invite**: Record granting a named user access to a project they do not own.
- **Pareto_Front**: Set of non-dominated solutions in a multi-objective optimization.
- **Utilization_Ratio**: Ratio of applied demand to code-allowable capacity (≤ 1.0 = pass).

### Requirement 1: Design System Consistency

**User Story:** As a user, I want every page in the platform to look and feel like a single
cohesive product, so that I can navigate confidently without encountering inconsistent styles
or broken layouts.

#### Acceptance Criteria

1. THE Platform SHALL apply a single shared design token set (colors, typography, spacing,
   border-radius, shadow) across all pages: LandingPage, Dashboard, SpacePlanningPage,
   ModernModeler, ReportBuilderPage, DetailingDesignPage, PricingPage, and all modal dialogs.
2. WHEN a user resizes the browser window to any width between 375px and 1920px, THE Platform
   SHALL render all pages without horizontal overflow, overlapping elements, or truncated
   primary actions.
3. THE Platform SHALL use a consistent component library for all interactive controls: buttons,
   inputs, selects, checkboxes, sliders, and tooltips SHALL share the same visual style across
   all pages.
4. WHEN a page is loading data, THE Platform SHALL display a skeleton loader or spinner that
   matches the layout of the content being loaded, rather than a blank panel.
5. WHEN a data fetch fails, THE Platform SHALL display an inline error state with a descriptive
   message and a retry action, rather than a blank panel or unhandled exception.
6. WHEN a list or table has no items to display, THE Platform SHALL render an empty-state
   illustration with a contextual message and a primary action (e.g., "Create your first
   project").
7. THE Platform SHALL use smooth CSS transitions (duration 150–300ms) for panel open/close,
   modal appear/dismiss, and tab switching, with no jarring layout shifts.
8. WHEN a user navigates between pages, THE Platform SHALL preserve scroll position within
   the same session for list pages (Dashboard, ReportBuilderPage).

### Requirement 4: House Plan Generator — Architectural Placement Verification

**User Story:** As an architect, I want the house plan generator to produce layouts that follow
real architectural logic, so that the generated plans are usable as a starting point for actual
design work.

#### Acceptance Criteria

1. WHEN `generateFloorPlan()` is called with a program containing at least one bedroom and one
   living room, THE SpacePlanningEngine SHALL place every bedroom such that its distance from
   the road-facing plot boundary is greater than the living room's distance from that boundary.
2. WHEN `generateFloorPlan()` is called with a program containing three or more habitable rooms,
   THE SpacePlanningEngine SHALL reserve a contiguous corridor zone of minimum clear width
   1.0 m along the central axis of the buildable envelope before placing any habitable room.
3. WHEN `generateFloorPlan()` is called with a program containing a kitchen or bathroom,
   THE SpacePlanningEngine SHALL position every wet area (kitchen, bathroom, toilet, utility,
   laundry) on a shared internal wall with at least one other wet area, unless the buildable
   width is less than the sum of the minimum wet-area widths.
4. WHEN `generateFloorPlan()` sizes any room from its area target, THE SpacePlanningEngine
   SHALL enforce that `room.width >= NBC_MIN_DIMS[room.type].w` AND
   `room.height >= NBC_MIN_DIMS[room.type].h` before computing the complementary dimension.
5. WHEN `generateFloorPlan()` is called with a program containing a living room and bedrooms,
   THE SpacePlanningEngine SHALL enforce the entrance sequence: foyer (if present) → living
   area → circulation spine → private zone, measured from the road-facing plot boundary.
6. WHEN `generateFloorPlan()` is called with a defined front-road orientation, THE
   SpacePlanningEngine SHALL apply vastu zone placement: kitchen in the SE quadrant, master
   bedroom in the SW quadrant, pooja room in the NE quadrant, living room in the N/NE quadrant.
7. WHEN the user's room list is missing any NBC-mandatory element (habitable room, kitchen,
   toilet), THE SpacePlanningEngine SHALL auto-inject the missing room at minimum NBC dimensions
   and SHALL record each injection in `FloorPlan.constraintViolations` with severity `"info"`.
8. WHEN `generateFloorPlan()` is called with any input, THE SpacePlanningEngine SHALL continue
   to ensure every PlacedRoom satisfies all setback constraints and no two PlacedRooms overlap
   by more than 0.01 m² (preservation of existing geometric correctness).

### Requirement 5: House Plan Generator — Multi-Floor and Plot Orientation Accuracy

**User Story:** As an architect, I want multi-floor plans and different plot orientations to
produce coherent layouts, so that the generator is useful for real projects beyond a single
ground-floor plan.

#### Acceptance Criteria

1. WHEN `generateFloorPlan()` is called with `floor > 0`, THE SpacePlanningEngine SHALL filter
   room specs to only those assigned to that floor and SHALL auto-inject parking only on floor 0.
2. WHEN a multi-floor project is generated, THE SpacePlanningEngine SHALL ensure that the
   structural grid columns on upper floors align with the column positions on the ground floor
   within a tolerance of 0.15 m.
3. WHEN the plot orientation is set to East-facing or West-facing, THE SpacePlanningEngine
   SHALL rotate the entrance-side logic accordingly so that the foyer is placed on the
   road-facing boundary regardless of compass direction.
4. WHEN a corner plot is specified (two road-facing sides), THE SpacePlanningEngine SHALL
   place the main entrance on the primary road side and SHALL apply reduced setbacks on the
   secondary road side per NBC Part 4.
5. WHEN `generateFloorPlan()` is called with a plot narrower than 6 m, THE SpacePlanningEngine
   SHALL produce a single-loaded corridor layout (rooms on one side only) and SHALL record a
   constraint note in `FloorPlan.constraintViolations`.

### Requirement 6: House Plan Generator — Round-Trip Serialization

**User Story:** As a developer, I want the space planning project to serialize and deserialize
without data loss, so that saved projects reload with identical room positions and MEP data.

#### Acceptance Criteria

1. THE SpacePlanningEngine SHALL serialize a `HousePlanProject` to a JSON string using
   `JSON.stringify` and deserialize it back using `JSON.parse`.
2. FOR ALL valid `HousePlanProject` objects, serializing then deserializing SHALL produce an
   object where every PlacedRoom has identical `x`, `y`, `width`, and `height` values.
3. FOR ALL valid `HousePlanProject` objects, serializing then deserializing SHALL produce an
   object where every MEP fixture has identical `x`, `y`, and `roomId` values.
4. IF a serialized project string is malformed or missing required fields, THEN THE
   SpacePlanningEngine SHALL return a descriptive error rather than a partially constructed
   object.

### Requirement 7: Analysis Pipeline — End-to-End Routing Correctness

**User Story:** As a structural engineer, I want analysis jobs to be routed to the correct
backend automatically, so that I always get results without manually selecting a solver.

#### Acceptance Criteria

1. WHEN `useAnalysis` is called with a model containing fewer than 500 nodes and
   `analysisType === 'static'`, THE Analysis_Router SHALL route the job to the WASM_Solver
   and return `result.backend === 'wasm'`.
2. WHEN `useAnalysis` is called with a model containing 500 or more nodes and the Rust_API
   is available, THE Analysis_Router SHALL route the job to the Rust_API and return
   `result.backend === 'rust'`.
3. WHEN the Rust_API is unavailable, THE Analysis_Router SHALL automatically fall back to the
   Python_API without any change required in the calling component, and SHALL display a
   non-blocking toast indicating the fallback backend.
4. WHEN all backends are unavailable, THE Analysis_Router SHALL surface a descriptive error
   to the user with a retry button.
5. THE result returned by `useAnalysis` SHALL always conform to the `AnalysisResult` interface
   (containing `displacements`, `reactions`, `memberForces`, `backend`, `computeTimeMs`)
   regardless of which backend processed the request.
6. WHEN analysis is in progress, THE Platform SHALL display a progress indicator showing the
   current solver step sourced from the solver's progress events, updated at least every 2
   seconds.

### Requirement 8: Analysis Pipeline — Advanced Analysis Types

**User Story:** As a structural engineer, I want P-Delta, buckling, modal, response spectrum,
pushover, and time history analyses to produce correct results visible in the results panels,
so that I can perform code-compliant design checks.

#### Acceptance Criteria

1. WHEN modal analysis results are returned, THE ModeShapeRenderer SHALL receive and render
   the mode shape data for all computed modes with natural frequencies and mass participation
   ratios displayed.
2. WHEN buckling analysis results are returned, THE StabilityView SHALL display the buckling
   load factors and mode shapes for all computed buckling modes.
3. WHEN response spectrum analysis is complete, THE Platform SHALL display combined base shear,
   story forces, and maximum member forces in the results panel.
4. WHEN pushover analysis is complete, THE Platform SHALL display the capacity curve (base
   shear vs. roof displacement) and the performance point.
5. WHEN load combination results are available, THE LoadCombosView SHALL consistently populate
   all combination result rows from the analysis result object.
6. WHEN design results are available, THE DCRatioView SHALL display utilization ratios for
   both steel members (from Rust/WASM backend) and concrete members (from Python backend).
7. WHEN steel design results are available, THE SteelDesignTab SHALL display results for both
   AISC 360-16 and IS 800:2007 design codes when results exist for either code.
8. WHEN RC design results are returned from the Python backend, THE RCBeamTab SHALL correctly
   map all result fields from the Python response schema to the UI component fields.

### Requirement 9: Report Generation — Data Accuracy

**User Story:** As a structural engineer, I want every generated report to contain accurate
data from the live model and user session, so that the report is suitable for professional
submission.

#### Acceptance Criteria

1. WHEN `handleExportPDF()` is called, THE Platform SHALL read the project name from the
   active project store and the engineer name from the authenticated user session, and SHALL
   pass these values to the report generator instead of hardcoded strings.
2. WHEN `generateBasicPDFReport` builds the analysis results summary table, THE Platform SHALL
   convert the maximum displacement value to millimetres (`* 1000`) before writing it to the
   summary row, and the unit label SHALL read "mm".
3. WHEN `generateBasicPDFReport` builds the member forces dictionary, THE Platform SHALL
   preserve the signed values for both `momentY` and `momentZ` at both ends of each member,
   so the report reflects the actual moment distribution.
4. WHEN `generateBasicPDFReport` generates a report, THE Platform SHALL include: a load cases
   section listing all applied load cases, a support reactions section listing all support
   reaction components, and a node displacement table listing all nodal displacements.
5. WHEN `generateBasicPDFReport` generates the steel design section, THE Platform SHALL display
   human-readable check descriptions (e.g., "Combined Compression + Flexure") and the
   applicable design code clause reference (e.g., "AISC 360-16 §H1-1" or "IS 800:2007 §9.3.1").
6. WHEN `ProfessionalReportGenerator` renders the report preview, THE Platform SHALL read node
   count, member count, storey count, and total height from `useModelStore` rather than from
   hardcoded placeholder values.
7. WHEN `generateQualityChecks()` is called, THE Platform SHALL derive pass/fail status from
   the actual `analysisResults` and `designResults` passed in, including real drift values,
   real deflection values, and real member utilization ratios.
8. WHEN `transformToDetailedReportData()` is called and `analysisResults` is not provided,
   THE Platform SHALL use `null` or `"N/A"` for all result fields rather than hardcoded
   fallback numbers.

### Requirement 16: Error Boundaries and Resilience

**User Story:** As a user, I want the application to recover gracefully from unexpected errors,
so that a crash in one panel does not destroy my entire work session.

#### Acceptance Criteria

1. WHEN `StructuralModelingCanvas` (Three.js) throws an uncaught exception, THE Platform SHALL
   catch it in an ErrorBoundary and render a "Reload Canvas" fallback UI without crashing the
   rest of the application.
2. WHEN `AnalysisDesignPanel` or `AIArchitectPanel` throws an uncaught exception, THE Platform
   SHALL catch it in an ErrorBoundary and render a graceful fallback UI without crashing the
   rest of the application.
3. WHEN `SpacePlanningPage` or any of its child panels throws an uncaught exception, THE
   Platform SHALL catch it in an ErrorBoundary and render a fallback UI with a "Reload" button.
4. WHEN the WASM solver fails, THE Analysis_Router SHALL log a warning and fall through to the
   Rust_API rather than surfacing an unhandled exception to the user.
5. WHEN the Python_API job queue times out (> 2 minutes), THE Analysis_Router SHALL surface a
   descriptive error to the user with a retry button.

### Requirement 17: Backend Health and Validation

**User Story:** As a DevOps engineer, I want all backend services to expose health check
endpoints and validate inputs, so that infrastructure monitoring works and malformed requests
are rejected cleanly.

#### Acceptance Criteria

1. THE Rust_API SHALL expose `GET /health` returning `{ "status": "ok", "version": "<semver>" }`
   with HTTP 200.
2. THE Python_API SHALL expose `GET /health` returning `{ "status": "ok", "version": "<semver>" }`
   with HTTP 200.
3. THE Node_API SHALL expose `GET /health` returning `{ "status": "ok", "version": "<semver>",
   "db": "connected" | "disconnected" }` with HTTP 200 when healthy and HTTP 503 when the
   database is unreachable.
4. ALL POST and PATCH routes in the Node_API SHALL validate `req.body` against a Zod schema
   before executing handler logic, returning HTTP 400 with `{ error: 'VALIDATION_ERROR',
   fields: [...] }` on failure.
5. THE Python_API analysis endpoints SHALL validate all incoming structural model payloads
   using Pydantic validators, rejecting models with node coordinates exceeding ±10,000 m,
   load magnitudes exceeding 1×10⁹ kN, or members referencing non-existent node IDs.
6. THE Node_API analysis proxy SHALL enforce per-tier model size limits before forwarding to
   the Python_API: free tier ≤ 100 nodes, pro tier ≤ 2,000 nodes, enterprise tier ≤ 10,000
   nodes; exceeding the limit returns HTTP 400 with `MODEL_TOO_LARGE`.

### Requirement 18: Payment and Billing Integrity

**User Story:** As a business owner, I want the billing system to be secure and idempotent,
so that users are never double-charged and revenue is correctly captured.

#### Acceptance Criteria

1. WHEN a PhonePe webhook is received, THE Node_API SHALL check whether a Subscription
   document with the same `phonepeMerchantTransactionId` already exists before processing,
   and IF a duplicate is detected SHALL return HTTP 200 without modifying any record.
2. THE PhonePe checkout initiation route SHALL derive the `amount` field exclusively from
   `BILLING_PLANS[planId].amountPaise` and SHALL NOT accept an `amount` field from the client.
3. WHEN a user's tier changes for any reason, THE Node_API SHALL write a record to the
   `TierChangeLog` collection containing `userId`, `fromTier`, `toTier`, `reason`, `timestamp`,
   and `transactionId` (if applicable).
4. WHEN a user initiates checkout with plan ID `business_monthly` or `business_yearly`, THE
   Node_API SHALL resolve the correct PhonePe amount and duration for the business plan.
5. IF a checkout plan ID does not match any entry in the billing config, THEN THE Node_API
   SHALL return HTTP 400 with a descriptive error message.

## Correctness Properties

*Properties are universal behavioral guarantees that hold across all valid inputs. They serve
as the bridge between human-readable requirements and machine-verifiable correctness.*

---

### Property 1: House Plan Zone Separation (Invariant)

*For any* `FloorPlanGenerationInput` containing at least one bedroom and one living room, the
fixed `generateFloorPlan` SHALL produce a layout where every bedroom's distance from the
road-facing plot boundary is strictly greater than the living room's distance from that boundary.

Formally: `∀ bedroom ∈ plan.rooms, ∀ living ∈ plan.rooms: distanceFromEntrance(bedroom) > distanceFromEntrance(living)`

**Validates: Requirement 4.1**

---

### Property 2: House Plan Circulation Spine (Invariant)

*For any* `FloorPlanGenerationInput` containing three or more habitable rooms, the fixed
`generateFloorPlan` SHALL produce a layout containing at least one corridor room with
`corridor.width >= 1.0` and every habitable room sharing at least one wall with that corridor.

**Validates: Requirement 4.2**

---

### Property 3: NBC Minimum Dimensions (Invariant)

*For any* `FloorPlanGenerationInput`, the fixed `generateFloorPlan` SHALL produce a layout
where every placed room satisfies `room.width >= NBC_MIN_DIMS[room.type].w` AND
`room.height >= NBC_MIN_DIMS[room.type].h`.

**Validates: Requirement 4.4**

---

### Property 4: House Plan Round-Trip Serialization (Round-Trip)

*For all* valid `HousePlanProject` objects P, `deserialize(serialize(P))` SHALL produce an
object where every `PlacedRoom` has identical `x`, `y`, `width`, `height` values and every
MEP fixture has identical `x`, `y`, `roomId` values to the original.

Formally: `∀ P: parse(format(P)) ≡ P` (round-trip property)

**Validates: Requirement 6.2, 6.3**

---

### Property 5: Analysis Routing by Node Count (Metamorphic)

*For any* `AnalysisModel` with `nodeCount < 500` and `analysisType === 'static'`, when a
WASM runner is available, `useAnalysis` SHALL return `result.backend === 'wasm'`.

*For any* `AnalysisModel` with `nodeCount >= 500`, when the Rust_API is available,
`useAnalysis` SHALL return `result.backend === 'rust'`.

**Validates: Requirement 7.1, 7.2**

---

### Property 6: Analysis Result Shape Invariant (Invariant)

*For any* valid `AnalysisModel`, the result returned by `useAnalysis` SHALL always conform to
the `AnalysisResult` interface — containing `displacements`, `reactions`, `memberForces`,
`backend`, and `computeTimeMs` — regardless of which backend processed the request.

**Validates: Requirement 7.5**

---

### Property 7: Report Uses Actual Project Data (Metamorphic)

*For any* `(projectName, engineerName)` pair where both are non-empty strings, the generated
PDF report SHALL contain `projectName` and `engineerName` in the cover page, and SHALL NOT
contain the strings `"BeamLab Project"` or `"Engineer"` as literal hardcoded values.

Formally: `∀ (name, eng): reportContains(report, name) ∧ reportContains(report, eng) ∧ ¬reportContains(report, "BeamLab Project")`

**Validates: Requirement 9.1**

---

### Property 8: Displacement Unit Conversion (Invariant)

*For any* analysis result with displacements in metres, the maximum displacement value written
to the PDF summary row SHALL equal `max(|dx|, |dy|, |dz|) * 1000` (converted to mm), with
absolute error less than 0.001 mm.

**Validates: Requirement 9.2**

---

### Property 9: Quality Checks Reflect Actual Results (Metamorphic)

*For any* `(analysisResults, designResults)` pair where both are non-null, the output of
`generateQualityChecks(analysisResults, designResults)` SHALL satisfy:
- `driftCheck.actual === analysisResults.maxDrift`
- `deflectionCheck.actual === analysisResults.maxDisplacement`
- `memberCheck.status` is derived from `designResults` utilization ratios

**Validates: Requirement 9.7**

---

### Property 10: Tier Gating Consistency (Invariant)

*For any* feature F where `TIER_CONFIG.free[F]` is `false` or `0`, when a free-tier user
interacts with a UI element gating feature F, the UpgradeModal SHALL be rendered and the
feature panel SHALL NOT be rendered.

Formally: `∀ F ∈ gatedFeatures, tier = 'free': upgradeModalVisible(F) = true ∧ featurePanelVisible(F) = false`

**Validates: Requirement 11.1**

---

### Property 11: canAccess Consistency (Invariant)

*For any* feature F and tier T, `canAccess(F)` called via the Subscription_Provider SHALL
return the same boolean as reading `TIER_CONFIG[T][F]` directly.

Formally: `∀ F, T: canAccess(F, T) = TIER_CONFIG[T][F]`

**Validates: Requirement 11.6**

---

### Property 12: Favorites Filter (Invariant)

*For any* collection of projects with mixed `isFavorited` and `deletedAt` values, the
Favorites tab query SHALL return only projects where `isFavorited === true` AND
`deletedAt === null`.

**Validates: Requirement 12.4**

---

### Property 13: Soft-Delete Exclusion (Invariant)

*For any* collection of projects with mixed `deletedAt` values, the default "My Projects"
query SHALL return only projects where `deletedAt === null`.

**Validates: Requirement 12.6**

---

### Property 14: Payment Idempotency (Idempotence)

*For any* `phonepeMerchantTransactionId` T, processing the same webhook twice SHALL produce
the same database state as processing it once — no duplicate Subscription records, no
duplicate tier upgrades.

Formally: `process(T) = process(process(T))` (idempotence)

**Validates: Requirement 18.1**

---

### Property 15: Payment Amount Server-Side Derivation (Invariant)

*For any* valid `planId` in `{pro_monthly, pro_yearly, business_monthly, business_yearly}`,
`BILLING_PLANS[planId].amountPaise` SHALL be greater than zero and `durationDays` SHALL be
greater than zero, and the checkout route SHALL use this value regardless of any client-
provided amount.

**Validates: Requirement 18.2**

---

### Property 16: Invalid Plan IDs Return HTTP 400 (Error Condition)

*For any* string that is not a member of `{pro_monthly, pro_yearly, business_monthly,
business_yearly}`, calling the checkout initiation endpoint with that plan ID SHALL return
HTTP 400.

**Validates: Requirement 18.5**

---

### Property 17: Boundary Enforcement Preservation (Invariant)

*For any* `FloorPlanGenerationInput` (including inputs that do NOT trigger the architectural
placement bug condition), every `PlacedRoom` in the output SHALL satisfy:
`room.x >= setback.left`, `room.y >= setback.front`,
`room.x + room.width <= plot.width - setback.right`,
`room.y + room.height <= plot.depth - setback.rear`.

**Validates: Requirement 4.8**

---

### Property 18: Pareto Front Non-Dominance (Invariant)

*For any* set of optimization solutions S, the Pareto_Front computed by the
SensitivityOptimizationDashboard SHALL contain only non-dominated solutions — no solution in
the front SHALL be strictly worse than another solution in the front on all objectives.

Formally: `∀ a, b ∈ ParetoFront: ¬(a dominates b) ∧ ¬(b dominates a)`

**Validates: Requirement 14.3**
