# Requirements Document

## Introduction

This spec captures the UX, documentation, onboarding, error handling, modeling workflow, and
monetization conversion requirements for BeamLab Ultimate, derived from a deep research pass
comparing BeamLab against SkyCiv and RISA-3D.

The research identified four critical gaps not covered by existing specs:

1. **Documentation and tutorials** — no public-facing tutorials, help articles, or video
   walkthroughs exist, making BeamLab invisible to new users and impossible to evaluate.
2. **Onboarding and time-to-first-analysis** — no guided first-run experience exists; new
   users land in the modeler with no orientation.
3. **Error handling UX** — no pre-analysis model validation, no actionable warning log, and
   no guided troubleshooting comparable to RISA-3D's Warning Log or SkyCiv's Repair Model.
4. **Conversion funnel** — no contextual in-app upgrade prompts tied to specific usage events
   (hitting storage limits, trying gated features, exporting with watermarks).

Billing enforcement, tier gating logic, analysis pipeline routing, and component-level UI
polish are covered in the beamlab-improvement-roadmap and beamlab-platform-refinement specs
and are explicitly out of scope here.

## Glossary

- **Onboarding_Flow**: The guided first-run experience shown to a user immediately after
  account creation, walking them through creating and analysing their first model.
- **Help_Center**: The in-app and public-facing documentation hub containing articles,
  tutorials, and video walkthroughs.
- **Interactive_Tutorial**: A step-by-step in-app walkthrough that highlights UI elements
  and prompts the user to perform specific actions in the live modeler.
- **Warning_Log**: A pre-analysis validation panel listing all model errors and warnings with
  actionable descriptions and links to the offending element.
- **Repair_Model**: An automated pre-analysis check that detects and optionally fixes common
  model issues (disconnected nodes, duplicate members, zero-length members).
- **Conversion_Prompt**: A contextual in-app message shown when a free-tier user reaches a
  usage limit or attempts a gated action, presenting a specific upgrade CTA.
- **Watermark**: A "BeamLab Free" overlay applied to PDF reports generated by free-tier users.
- **Free_Tier**: The unpaid plan with public-only projects, limited project count, model size
  limits, and watermarked reports.
- **Pro_Tier**: The paid professional plan with private projects, full analysis, integrated
  design modules, and unbranded reports.
- **Business_Tier**: The paid team plan with all Pro features plus collaboration and priority
  support.
- **Modeler**: The `ModernModeler.tsx` workspace where users build and analyse structural models.
- **Dashboard**: The `Dashboard.tsx` page listing the user's projects.
- **AI_Assistant**: The AI design assistant feature accessible from the Modeler.
- **Section_Library**: The integrated database of standard steel and concrete section profiles
  filterable by country and design code.
- **Load_Combination_Generator**: The tool that automatically generates load combinations per
  a selected design code (ASCE 7, IS 875, Eurocode).
- **Results_Viewer**: The panel displaying colour-coded force diagrams, deflected shapes, and
  numerical results after analysis.
- **Calculation_Report**: The step-by-step design check output showing governing limit states,
  demand/capacity ratios, and code clause references.

## Requirements

---

### Requirement 1: Public Documentation and Help Center

**User Story:** As a new user evaluating BeamLab, I want to find tutorials and documentation
before signing up, so that I can assess whether the product meets my needs without committing
to an account.

#### Acceptance Criteria

1. THE Help_Center SHALL be publicly accessible at a stable URL (e.g., `/docs` or a subdomain)
   without requiring authentication.
2. THE Help_Center SHALL contain at minimum the following articles at launch: Getting Started
   (account creation and first model), Modeler Overview (toolbar sections and navigation),
   Running Your First Analysis, Interpreting Results, Generating a Report, and Pricing and
   Plan Comparison.
3. THE Help_Center SHALL contain at minimum one video walkthrough (screen recording with
   narration) covering the end-to-end workflow: model creation → load assignment → analysis →
   report export.
4. WHEN a user searches the Help_Center, THE Help_Center SHALL return matching articles within
   2 seconds and SHALL display a "No results found" message with suggested related articles
   when no match exists.
5. THE Help_Center SHALL include a Glossary page defining all domain terms used in the UI
   (e.g., node, member, DOF, load combination, utilization ratio).
6. WHEN a new feature is released, THE Help_Center SHALL be updated with a corresponding
   article or section within 14 days of the feature's public availability.

### Requirement 2: In-App Contextual Help

**User Story:** As a user working in the Modeler, I want contextual help available without
leaving the workspace, so that I can resolve questions without breaking my flow.

#### Acceptance Criteria

1. THE Modeler SHALL display a persistent help icon (?) in the toolbar that opens a contextual
   help panel showing the article relevant to the currently active tool or panel.
2. WHEN a user hovers over any toolbar tool for more than 1 second, THE Modeler SHALL display
   a tooltip containing the tool name, a one-sentence description, and a keyboard shortcut
   if one exists.
3. THE Modeler SHALL display a "What's this?" link in every dialog header that opens the
   corresponding Help_Center article in a new browser tab.
4. WHEN a user opens the Modeler for the first time, THE Modeler SHALL display a dismissible
   welcome banner with a "Take the tour" CTA that launches the Interactive_Tutorial.
5. THE Help_Center SHALL be linked from the Dashboard navigation, the Modeler toolbar, and
   the account settings page.

### Requirement 3: Guided Onboarding Flow

**User Story:** As a new user who has just created an account, I want a guided first-run
experience, so that I can complete my first structural analysis within 15 minutes without
external help.

#### Acceptance Criteria

1. WHEN a user completes account creation for the first time, THE Platform SHALL redirect the
   user to the Onboarding_Flow before showing the Dashboard.
2. THE Onboarding_Flow SHALL consist of at most five steps: (1) role selection (student,
   practising engineer, researcher, other), (2) primary use case selection (steel frame,
   RC frame, truss, other), (3) unit system preference (SI / Imperial), (4) a pre-built
   example model loaded in the Modeler, (5) a prompt to run analysis on the example model.
3. WHEN the user completes step 5 of the Onboarding_Flow, THE Platform SHALL display a
   success screen showing the analysis results and a CTA to "Start your own project".
4. WHEN the user dismisses the Onboarding_Flow at any step, THE Platform SHALL save the
   completed steps and SHALL NOT repeat completed steps if the user re-enters the flow.
5. THE Onboarding_Flow SHALL be re-accessible from the Dashboard via a "Restart tutorial"
   link in the account settings page.
6. WHEN the user completes the Onboarding_Flow, THE Platform SHALL record the completion
   event with a timestamp so that product analytics can measure time-to-first-analysis.

### Requirement 4: Interactive Tutorial System

**User Story:** As a user learning the Modeler, I want step-by-step in-app guidance that
highlights the relevant UI element at each step, so that I can learn by doing rather than
reading a separate manual.

#### Acceptance Criteria

1. THE Interactive_Tutorial SHALL support at minimum the following tutorial sequences:
   "Create a simple beam", "Add supports and loads", "Run static analysis", and
   "Export a PDF report".
2. WHEN an Interactive_Tutorial step is active, THE Platform SHALL highlight the target UI
   element with a visual overlay and SHALL display a tooltip with the step instruction and
   step counter (e.g., "Step 3 of 8").
3. WHEN the user completes the action required by a tutorial step, THE Platform SHALL
   automatically advance to the next step without requiring the user to click "Next".
4. WHEN the user clicks "Skip tutorial" at any step, THE Platform SHALL dismiss the tutorial
   and SHALL NOT show it again unless the user explicitly restarts it.
5. THE Interactive_Tutorial SHALL be accessible from the Help_Center and from the Modeler
   help panel at any time, not only during onboarding.
6. IF the user navigates away from the Modeler while a tutorial is active, THEN THE Platform
   SHALL pause the tutorial and SHALL offer to resume it when the user returns to the Modeler.

### Requirement 5: Pre-Analysis Model Validation (Warning Log)

**User Story:** As a structural engineer, I want the platform to check my model for errors
before I run analysis, so that I receive actionable feedback rather than a cryptic solver
failure.

#### Acceptance Criteria

1. WHEN the user clicks "Run Analysis", THE Platform SHALL execute a pre-analysis validation
   pass before submitting the model to any solver.
2. THE pre-analysis validation SHALL detect and report the following conditions as errors
   (blocking analysis): zero-length members, members referencing non-existent nodes, nodes
   with no connected members, models with no supports defined, and models with no loads
   defined.
3. THE pre-analysis validation SHALL detect and report the following conditions as warnings
   (non-blocking): members with very high aspect ratios (length/depth > 500), nodes that are
   geometrically close but not connected (within 0.01 m), duplicate member definitions
   (same start and end node), and load cases with no applied loads.
4. WHEN validation produces errors or warnings, THE Warning_Log panel SHALL open automatically
   and SHALL list each issue with: a severity icon (error/warning), a human-readable
   description (e.g., "Member M12 references non-existent node N99"), and a "Go to" button
   that selects and centres the offending element in the viewport.
5. WHEN the user clicks "Go to" for a Warning_Log entry, THE Modeler SHALL select the
   offending element, centre the viewport on it, and highlight it in red (error) or amber
   (warning).
6. IF the model contains any errors, THEN THE Platform SHALL block analysis submission and
   SHALL display the count of blocking errors in the "Run Analysis" button label
   (e.g., "Fix 3 errors to run").
7. WHEN all errors are resolved, THE Platform SHALL re-run the validation pass automatically
   and SHALL update the Warning_Log and the "Run Analysis" button state within 500ms.
8. THE Warning_Log SHALL remain accessible after analysis is complete so that the user can
   review warnings alongside results.

### Requirement 6: Automated Model Repair

**User Story:** As a structural engineer, I want the platform to offer automatic fixes for
common model issues, so that I can resolve simple problems without manually hunting for
disconnected nodes.

#### Acceptance Criteria

1. THE Warning_Log SHALL include a "Repair Model" button that triggers the Repair_Model
   function.
2. WHEN Repair_Model is triggered, THE Platform SHALL automatically fix the following
   issues if present: merge nodes within 0.01 m of each other, remove zero-length members,
   and remove exact duplicate member definitions.
3. WHEN Repair_Model completes, THE Platform SHALL display a summary notification listing
   each type of fix applied and the count of elements affected (e.g., "Merged 3 duplicate
   nodes, removed 1 zero-length member").
4. WHEN Repair_Model makes changes, THE Platform SHALL record the operation in the model's
   undo history so the user can revert the repair with a single Ctrl+Z.
5. IF Repair_Model finds no fixable issues, THEN THE Platform SHALL display a notification
   stating "No automatic repairs needed."

### Requirement 7: Interactive Results Visualization

**User Story:** As a structural engineer, I want to explore analysis results interactively in
the viewport, so that I can understand the structural behaviour without switching to a separate
results table.

#### Acceptance Criteria

1. WHEN analysis results are available, THE Results_Viewer SHALL display colour-coded force
   diagrams (bending moment, shear, axial) overlaid on the structural model in the viewport,
   with a colour scale legend showing the min and max values and their units.
2. WHEN the user hovers over any point on a force diagram, THE Results_Viewer SHALL display
   a tooltip showing the exact force value, the member ID, and the fractional position along
   the member.
3. WHEN the user selects "Deflected Shape" in the Results_Viewer, THE Platform SHALL animate
   the structure deforming from its undeformed position to its deflected position at a
   user-adjustable scale factor (default 100×).
4. THE Results_Viewer SHALL allow the user to toggle individual result types (moment, shear,
   axial, deflection) on and off independently.
5. WHEN the user selects a member in the Results_Viewer, THE Platform SHALL display a
   side panel showing the full force diagram for that member across its length, with values
   at the start node, midpoint, and end node.
6. THE Results_Viewer SHALL display support reactions as labelled arrows at each support node,
   with the reaction magnitude and direction shown in the tooltip on hover.
7. WHEN the user switches between load cases or combinations in the Results_Viewer, THE
   viewport SHALL update the force diagrams within 300ms without requiring a full re-render
   of the model geometry.

### Requirement 8: Transparent Design Check Reports

**User Story:** As a structural engineer, I want to see step-by-step design calculations for
each member, so that I can verify the software's output and use it in a professional submission.

#### Acceptance Criteria

1. WHEN design results are available for a member, THE Platform SHALL provide a
   Calculation_Report for that member accessible from the Results_Viewer by clicking the
   member.
2. THE Calculation_Report SHALL show, for each governing limit state checked (e.g., flexure,
   shear, combined axial + flexure): the applied demand value, the code-allowable capacity
   value, the utilization ratio (demand / capacity), the governing load combination, and the
   applicable code clause reference (e.g., "AISC 360-16 §H1-1").
3. THE Calculation_Report SHALL identify the controlling (highest utilization ratio) limit
   state and SHALL display it at the top of the report with a pass/fail badge.
4. WHEN the user clicks "Export Calculation Sheet" in the Calculation_Report, THE Platform
   SHALL generate a PDF containing the full step-by-step calculation for that member,
   formatted for professional submission.
5. THE Calculation_Report SHALL be included as an appendix in the full project PDF report
   when the user selects "Include design calculations" in the report export options.
6. WHEN a member fails a design check (utilization ratio > 1.0), THE Results_Viewer SHALL
   highlight that member in red in the viewport and SHALL display the controlling utilization
   ratio as a label on the member.

### Requirement 9: Professional Report Generation

**User Story:** As a structural engineer, I want to generate a professional PDF report with
a customizable title block, so that I can submit it directly to a client or authority without
post-processing.

#### Acceptance Criteria

1. THE Platform SHALL provide a report export dialog that allows the user to configure:
   project name, engineer name, company name, project number, date, and a logo image upload.
2. WHEN the user uploads a logo image, THE Platform SHALL display it in the report title block
   and SHALL persist it for future reports in the same project.
3. THE Platform SHALL allow the user to select which sections to include in the report:
   project information, model summary, load cases, load combinations, node displacements,
   support reactions, member forces, design check summary, and individual calculation sheets.
4. WHEN a free-tier user generates a PDF report, THE Platform SHALL apply a "BeamLab Free"
   watermark on every page of the exported PDF.
5. WHEN a Pro-tier or Business-tier user generates a PDF report, THE Platform SHALL produce
   an unbranded PDF with no watermark.
6. THE Platform SHALL generate the PDF report within 10 seconds for models with up to 500
   members.
7. WHEN the PDF is generated, THE Platform SHALL offer both a direct download and a "Copy
   shareable link" option that generates a time-limited (7-day) public URL for the report.

### Requirement 10: Contextual Conversion Prompts

**User Story:** As a product manager, I want free-tier users to see specific, relevant upgrade
prompts at the moment they hit a limit, so that the upgrade value proposition is clear and
timely.

#### Acceptance Criteria

1. WHEN a free-tier user attempts to save a project and has reached the maximum free-tier
   project count, THE Platform SHALL display a Conversion_Prompt explaining the limit
   (e.g., "You've used all 3 free projects") and presenting a Pro plan CTA with the monthly
   price.
2. WHEN a free-tier user saves a project, THE Platform SHALL display a persistent notice
   stating "This project is public — anyone with the link can view it" and SHALL include an
   "Upgrade for private projects" link.
3. WHEN a free-tier user attempts to export a PDF report, THE Platform SHALL display a
   Conversion_Prompt explaining that the free export includes a watermark and presenting a
   Pro plan CTA to remove it.
4. WHEN a free-tier user attempts to access an integrated design module (AISC, ACI, Eurocode
   design checks), THE Platform SHALL display a Conversion_Prompt naming the specific module
   and presenting a Pro plan CTA.
5. WHEN a free-tier user's model exceeds the free-tier node or member count limit, THE
   Platform SHALL display a Conversion_Prompt before blocking the add operation, explaining
   the limit and presenting a Pro plan CTA.
6. WHEN a free-tier user attempts to use the AI_Assistant, THE Platform SHALL display a
   Conversion_Prompt explaining that AI assistance is a Pro feature and presenting a Pro
   plan CTA.
7. WHEN a Conversion_Prompt is displayed, THE Platform SHALL include a "Maybe later" dismiss
   option that closes the prompt without navigating away.
8. THE Platform SHALL record each Conversion_Prompt display event with the trigger type and
   user tier so that conversion funnel analytics can be computed.

### Requirement 11: Free Tier Strategic Limitations

**User Story:** As a product manager, I want the free tier to be genuinely useful but
strategically limited, so that users can evaluate the product and are motivated to upgrade
for professional use.

#### Acceptance Criteria

1. THE Free_Tier SHALL allow users to save up to 3 projects, all of which are publicly
   accessible via a shareable link.
2. THE Free_Tier SHALL limit structural models to a maximum of 100 nodes and 200 members.
3. THE Free_Tier SHALL allow static linear analysis only; P-Delta, buckling, modal, response
   spectrum, pushover, and time history analyses SHALL be gated to Pro_Tier and above.
4. THE Free_Tier SHALL allow PDF report export with a "BeamLab Free" watermark on every page.
5. THE Free_Tier SHALL NOT provide access to integrated design modules (AISC 360, ACI 318,
   Eurocode 3/4, IS 800, IS 456).
6. THE Free_Tier SHALL NOT provide access to the AI_Assistant.
7. THE Free_Tier SHALL NOT provide access to collaboration features.
8. WHEN a free-tier user views the Dashboard, THE Platform SHALL display the current usage
   counts (e.g., "2 of 3 projects used") alongside each applicable limit.

### Requirement 12: Modeling Efficiency — Dual Input Mode

**User Story:** As a structural engineer, I want to input model geometry using both a graphical
canvas and a spreadsheet-style table, so that I can choose the most efficient method for each
task.

#### Acceptance Criteria

1. THE Modeler SHALL provide a spreadsheet input panel (accessible via a toolbar toggle) that
   displays all nodes as rows with editable X, Y, Z coordinate columns.
2. THE Modeler SHALL provide a spreadsheet input panel for members displaying member ID, start
   node, end node, and section profile columns, all editable inline.
3. WHEN the user edits a node coordinate in the spreadsheet panel, THE Modeler SHALL update
   the 3D viewport geometry within 200ms without requiring a manual refresh.
4. WHEN the user edits a member definition in the spreadsheet panel, THE Modeler SHALL update
   the 3D viewport geometry within 200ms.
5. THE Modeler SHALL support grid snapping in the graphical canvas with a configurable grid
   spacing (default 0.5 m), toggleable via a toolbar button or the G key.
6. WHEN the user draws a node in the graphical canvas with grid snapping enabled, THE Modeler
   SHALL snap the node to the nearest grid intersection and SHALL display the snapped
   coordinates in the status bar.
7. THE Modeler SHALL support coordinate input via a command bar (similar to CAD software)
   where the user can type absolute coordinates (e.g., "3.0, 0.0, 4.5") or relative
   coordinates (e.g., "@1.5, 0, 0") to place nodes precisely.

### Requirement 13: Section Library and Load Generation

**User Story:** As a structural engineer, I want to select sections from a filterable library
and generate load combinations automatically, so that I can set up a model quickly without
manual data entry.

#### Acceptance Criteria

1. THE Section_Library SHALL be accessible from the member properties panel and SHALL contain
   standard profiles for AISC (W, S, C, L, HSS), IS (ISMB, ISMC, ISA, ISHB), and Eurocode
   (IPE, HEA, HEB, UB, UC) families.
2. WHEN the user opens the Section_Library, THE Platform SHALL allow filtering by country,
   design code, profile family, and a numeric range for depth or weight.
3. WHEN the user selects a section from the Section_Library, THE Platform SHALL display the
   key section properties (area, Ix, Iy, Sx, Sy, weight per metre) before the user confirms
   the selection.
4. THE Load_Combination_Generator SHALL be accessible from the loads panel and SHALL support
   the following codes: ASCE 7-22, IS 875 Part 5, and EN 1990.
5. WHEN the user selects a design code in the Load_Combination_Generator and confirms, THE
   Platform SHALL generate all required strength and serviceability load combinations for the
   defined load cases and SHALL add them to the model's load combination list.
6. WHEN the user activates "Apply Self-Weight", THE Platform SHALL compute the self-weight of
   each member from its section profile and material density and SHALL add it as a distributed
   load in the active gravity load case.

### Requirement 14: AI Assistant Documentation and Discoverability

**User Story:** As a user, I want to understand what the AI Assistant can do and how to use
it, so that I can take advantage of this differentiating feature.

#### Acceptance Criteria

1. THE Help_Center SHALL contain a dedicated "AI Assistant" article explaining: what the
   assistant can do (e.g., suggest section sizes, explain analysis results, generate load
   combinations), how to access it, example prompts, and known limitations.
2. WHEN the user opens the AI_Assistant panel for the first time, THE Platform SHALL display
   a brief onboarding tooltip listing three example prompts (e.g., "Suggest a section for
   this beam", "Explain this warning", "Generate ASCE 7 load combinations").
3. THE AI_Assistant panel SHALL display a persistent "How to use" link that opens the
   Help_Center AI Assistant article.
4. WHEN the AI_Assistant returns a suggestion that modifies the model (e.g., changes a
   section), THE Platform SHALL display a confirmation dialog showing the proposed change
   before applying it, and SHALL record the change in the undo history.
5. WHEN the AI_Assistant is unavailable due to a service error, THE Platform SHALL display
   a descriptive error message (e.g., "AI Assistant is temporarily unavailable") rather than
   a blank panel or unhandled exception.

### Requirement 15: Social Proof and Case Studies

**User Story:** As a prospective user evaluating BeamLab, I want to see real-world examples
of projects completed with the platform, so that I can assess its suitability for my work.

#### Acceptance Criteria

1. THE LandingPage SHALL include a case studies or example projects section containing at
   minimum three worked examples with: a project description, a screenshot of the model and
   results, the analysis type used, and the design code applied.
2. THE LandingPage SHALL include a testimonials section with at minimum three user quotes
   attributed to a role (e.g., "Structural Engineer, 8 years experience") — PII such as
   full names SHALL be replaced with role-based attributions unless explicit consent is
   obtained.
3. THE LandingPage SHALL display a trust bar showing key platform statistics (e.g., number
   of analyses run, number of registered users) sourced from a backend counter endpoint,
   updated at most once per hour.
4. WHEN the trust bar statistics endpoint is unavailable, THE LandingPage SHALL display the
   last cached values rather than a blank section.
5. THE LandingPage SHALL include a "Try a live example" CTA that loads a pre-built example
   model in the Modeler without requiring account creation, allowing anonymous users to
   experience the analysis workflow.

### Requirement 16: Conversion Funnel Analytics

**User Story:** As a product manager, I want to measure where users drop off in the conversion
funnel, so that I can prioritise UX improvements that increase paid conversions.

#### Acceptance Criteria

1. THE Platform SHALL record the following funnel events with a timestamp and user tier:
   `page_view_landing`, `signup_started`, `signup_completed`, `onboarding_completed`,
   `first_analysis_run`, `upgrade_prompt_shown` (with trigger type), `upgrade_cta_clicked`,
   `checkout_started`, `checkout_completed`.
2. WHEN any funnel event is recorded, THE Platform SHALL send it to the analytics endpoint
   within 5 seconds of the triggering action.
3. THE Node_API SHALL expose a `GET /api/admin/funnel-stats` endpoint (admin-only) that
   returns aggregate counts for each funnel event over a configurable date range.
4. THE Platform SHALL record funnel events without collecting any PII beyond the anonymised
   user ID and session ID.
5. WHEN a user has opted out of analytics tracking, THE Platform SHALL NOT record any funnel
   events for that user.
