# Requirements Document

## Introduction

This feature improves three interconnected tools in BeamLab:

1. **Space Planning Tool** (`SpacePlanningPage.tsx` / `SpacePlanningEngine.ts`) — fixes room placement accuracy so that solver-placed rooms always fit within the actual plot boundary, eliminates overlaps, and aligns MEP overlays with placed rooms. Elevation and section views are upgraded from placeholder quality to dimensioned architectural drawings.

2. **Sensitivity & Optimization Dashboard** (`SensitivityOptimizationDashboard.tsx`) — extends the optimizer beyond single-objective section-depth sweeps to support multi-objective Pareto-front exploration, adds a working parameter study tab, adds convergence visualization, and enables export of optimized sections back to the model store.

3. **Structural Detailing Center** (`DetailingDesignPage.tsx`) — replaces the static overview card grid with a live pass/fail status dashboard per member, adds batch design capability, enables reinforcement drawing export, and adds a design report generator.

---

## Glossary

- **SpacePlanningEngine**: The client-side TypeScript class in `SpacePlanningEngine.ts` that generates floor plans, structural grids, and MEP plans.
- **CSP_Solver**: The backend constraint-satisfaction solver accessed via `layoutApiService` that returns optimized room placements.
- **PlacedRoom**: A room object with resolved `x`, `y`, `width`, `height` coordinates in metres relative to the plot origin.
- **Plot_Boundary**: The rectangle defined by `plot.width × plot.depth` minus the four setbacks (`front`, `rear`, `left`, `right`).
- **Buildable_Envelope**: The inner rectangle after subtracting all setbacks from the Plot_Boundary.
- **MEP_Overlay**: The electrical, plumbing, or HVAC fixture layer rendered on top of the floor plan canvas.
- **Structural_Grid**: The column-and-beam grid generated by `generateStructuralPlan()`.
- **Variant**: One of up to five alternative layout solutions returned by `generateLayoutVariants()`.
- **Constraint_Report**: The `ConstraintReport` object produced by the CSP_Solver or `buildConstraintReportFromVariant()` containing per-domain pass/fail violations.
- **Sensitivity_Analysis**: Perturbation-based gradient estimation of how each design variable affects the objective function.
- **Pareto_Front**: The set of non-dominated solutions in a multi-objective optimization where no objective can be improved without worsening another.
- **Design_Variable**: A section property (depth, width, grade) assigned to a group of members that the optimizer may vary.
- **Utilization_Ratio**: The ratio of applied demand to code-allowable capacity for a structural member (≤ 1.0 = pass).
- **Batch_Design**: Running code checks on all members of a given type in a single operation.
- **Reinforcement_Drawing**: A dimensioned SVG or PDF sketch showing bar layout, curtailment, and spacing for a designed member.
- **Design_Report**: A structured PDF or HTML document summarizing all member design results, code references, and pass/fail status.
- **NBC**: National Building Code of India 2016.
- **IS_456**: Indian Standard for Plain and Reinforced Concrete (IS 456:2000).
- **IS_800**: Indian Standard for General Construction in Steel (IS 800:2007).

## Requirements

### Requirement 1: Room Placement Boundary Enforcement

**User Story:** As a structural engineer, I want every generated room to fit entirely within the buildable envelope, so that the floor plan is geometrically valid and ready for further detailing.

#### Acceptance Criteria

1. WHEN the CSP_Solver returns placements, THE SpacePlanningEngine SHALL validate that each PlacedRoom satisfies `room.x >= setback.left`, `room.y >= setback.front`, `room.x + room.width <= plot.width - setback.right`, and `room.y + room.height <= plot.depth - setback.rear`.
2. IF a PlacedRoom from the CSP_Solver violates the Buildable_Envelope, THEN THE SpacePlanningEngine SHALL clamp the room coordinates to the nearest valid position within the Buildable_Envelope and log a warning with the room id and the amount of correction applied.
3. WHEN the client-side `placeRooms()` fallback is used, THE SpacePlanningEngine SHALL reject any candidate position where `room.x + room.width > buildableWidth + setback.left` or `room.y + room.height > buildableDepth + setback.front`.
4. THE SpacePlanningEngine SHALL ensure that no two PlacedRooms share overlapping area, defined as the intersection of their axis-aligned bounding boxes having area greater than 0.01 m².
5. IF two PlacedRooms overlap after placement, THEN THE SpacePlanningEngine SHALL resolve the overlap by translating the lower-priority room along the axis of minimum penetration depth before finalising the floor plan.
6. WHEN a floor plan is finalised, THE SpacePlanningEngine SHALL compute and expose a `boundaryViolationCount` and `overlapCount` on the returned `FloorPlan` object so the UI can surface them in the Constraint_Report.

### Requirement 2: Adjacency-Aware Room Placement

**User Story:** As an architect, I want rooms to respect adjacency and avoidance rules during placement, so that the generated layout reflects real design intent (e.g. kitchen adjacent to dining, bathrooms away from living).

#### Acceptance Criteria

1. THE SpacePlanningEngine SHALL score each candidate room position using an adjacency score computed as the sum of shared-wall length with rooms listed in `room.adjacentTo`, minus the sum of shared-wall length with rooms listed in `room.awayFrom`.
2. WHEN placing a room, THE SpacePlanningEngine SHALL prefer positions with the highest adjacency score over positions with lower scores, all else being equal.
3. THE SpacePlanningEngine SHALL place bathrooms and toilets adjacent to at least one bedroom or master_bedroom when the floor plan contains bedrooms.
4. THE SpacePlanningEngine SHALL place the kitchen within two room-widths of the dining room when both are present on the same floor.
5. WHEN no valid position satisfies all adjacency preferences, THE SpacePlanningEngine SHALL place the room at the best available position and record an adjacency violation in the `FloorPlan.constraintViolations` array.

---

### Requirement 3: Structural Grid Alignment with Room Boundaries

**User Story:** As a structural engineer, I want the structural grid columns to align with room corners and wall intersections, so that the structural plan is architecturally coordinated.

#### Acceptance Criteria

1. WHEN `generateStructuralPlan()` is called, THE SpacePlanningEngine SHALL snap each column position to the nearest room corner within a tolerance of 0.15 m before placing it on the grid.
2. THE SpacePlanningEngine SHALL ensure that no column is placed inside the clear interior of any PlacedRoom (i.e. more than `STRUCTURAL_GRID.COLUMN_SIZE / 2` from all room walls).
3. THE SpacePlanningEngine SHALL generate beams along room boundary walls where the span between adjacent columns is between `STRUCTURAL_GRID.MIN_SPAN` and `STRUCTURAL_GRID.MAX_SPAN`.
4. WHEN a room wall does not align with any grid line within 0.3 m, THE SpacePlanningEngine SHALL insert an intermediate column at the wall midpoint if the resulting spans remain within `STRUCTURAL_GRID.MAX_SPAN`.
5. THE SpacePlanningEngine SHALL expose a `gridAlignmentScore` (0–100) on the `StructuralPlan` object, computed as the percentage of room corners that have a column within 0.15 m.

---

### Requirement 4: MEP Overlay Coordinate Alignment

**User Story:** As a building services engineer, I want electrical, plumbing, and HVAC fixtures to be positioned relative to the actual placed room coordinates, so that MEP overlays visually align with the floor plan.

#### Acceptance Criteria

1. WHEN `generateElectricalPlan()` is called, THE SpacePlanningEngine SHALL compute all fixture `x` and `y` coordinates as offsets from the corresponding PlacedRoom's `x` and `y` origin, not from the plot origin.
2. WHEN `generatePlumbingPlan()` is called, THE SpacePlanningEngine SHALL place wet fixtures (WC, basin, shower, sink) within the bounds of their assigned PlacedRoom.
3. WHEN `generateHVACPlan()` is called, THE SpacePlanningEngine SHALL place indoor AC units and ventilation paths within the bounds of their assigned PlacedRoom.
4. WHEN solver placements override engine room positions, THE SpacePlanningEngine SHALL recompute all MEP fixture coordinates using the updated PlacedRoom positions before returning the final project.
5. THE SpacePlanningEngine SHALL validate that every generated MEP fixture satisfies `fixture.x >= room.x` and `fixture.x <= room.x + room.width` and `fixture.y >= room.y` and `fixture.y <= room.y + room.height` for its assigned room, and SHALL log a warning for any fixture that fails this check.

### Requirement 5: Elevation and Section View Quality

**User Story:** As an architect, I want elevation and section views to show dimensioned outlines with window and door openings, so that the drawings are useful for client presentations and permit applications.

#### Acceptance Criteria

1. WHEN the elevations tab is rendered, THE ElevationSectionViewer SHALL draw the building outline to scale using the actual PlacedRoom heights and floor-to-floor heights from the floor plan.
2. THE ElevationSectionViewer SHALL render window and door openings on each elevation face at the correct horizontal position derived from the PlacedRoom coordinates and the door/window specs.
3. THE ElevationSectionViewer SHALL render at least two horizontal dimension lines per elevation: one showing the total building width and one showing individual room widths along that face.
4. THE ElevationSectionViewer SHALL render at least one vertical dimension line per elevation showing the floor-to-floor height and total building height.
5. WHEN a section line (A-A or B-B) is rendered, THE ElevationSectionViewer SHALL show the cut rooms in cross-section with slab thickness, beam depth, and floor-to-floor height labelled.
6. THE ElevationSectionViewer SHALL render a north arrow and a scale bar on every elevation and section drawing.

---

### Requirement 6: Multi-Objective Pareto Optimization

**User Story:** As a structural engineer, I want to explore trade-offs between weight, cost, and displacement in the optimizer, so that I can make informed section selection decisions.

#### Acceptance Criteria

1. WHEN the optimization tab is active and the user selects two or more objectives, THE SensitivityOptimizationDashboard SHALL compute a Pareto_Front by evaluating each candidate section combination against all selected objectives.
2. THE SensitivityOptimizationDashboard SHALL display the Pareto_Front as a scatter plot with one objective on each axis, where each point represents a feasible non-dominated solution.
3. WHEN the user clicks a point on the Pareto_Front scatter plot, THE SensitivityOptimizationDashboard SHALL highlight the corresponding section assignments in the design variables panel.
4. THE SensitivityOptimizationDashboard SHALL support at least the following objective pairs: (minimize-weight, minimize-displacement), (minimize-weight, minimize-cost), and (minimize-displacement, maximize-stiffness).
5. THE SensitivityOptimizationDashboard SHALL label each Pareto_Front point with its total weight in kg and maximum displacement in mm.
6. IF no feasible solutions exist for the selected constraints, THEN THE SensitivityOptimizationDashboard SHALL display a message identifying which constraint is infeasible and suggest relaxing it.

---

### Requirement 7: Convergence Visualization

**User Story:** As a structural engineer, I want to see how the optimizer converges over iterations, so that I can judge whether the result is reliable or needs more iterations.

#### Acceptance Criteria

1. WHEN an optimization run is in progress, THE SensitivityOptimizationDashboard SHALL update a line chart showing objective value versus iteration number in real time, with updates no less frequently than every 5 iterations.
2. WHEN an optimization run completes, THE SensitivityOptimizationDashboard SHALL display the final convergence chart with the best objective value annotated on the chart.
3. THE SensitivityOptimizationDashboard SHALL display a convergence indicator that shows "Converged" when the change in objective value over the last 10 iterations is less than the convergence tolerance, and "Not Converged" otherwise.
4. THE SensitivityOptimizationDashboard SHALL display the total number of function evaluations and elapsed time in seconds upon completion of the optimization run.

---

### Requirement 8: Parameter Study Tab

**User Story:** As a structural engineer, I want to run a parameter study that sweeps one or two design variables across a range, so that I can understand the sensitivity surface without running a full optimization.

#### Acceptance Criteria

1. WHEN the parameter study tab is active, THE SensitivityOptimizationDashboard SHALL allow the user to select one or two Design_Variables from the current model and specify a lower bound, upper bound, and number of steps for each.
2. WHEN the user runs a one-variable parameter study, THE SensitivityOptimizationDashboard SHALL evaluate the objective function at each step value and display the results as a line chart with the variable value on the x-axis and objective value on the y-axis.
3. WHEN the user runs a two-variable parameter study, THE SensitivityOptimizationDashboard SHALL evaluate the objective function at each combination of step values and display the results as a heat map with one variable on each axis and objective value encoded as colour.
4. THE SensitivityOptimizationDashboard SHALL complete a parameter study of up to 100 total evaluations within 10 seconds on a model with fewer than 500 members.
5. WHEN a parameter study completes, THE SensitivityOptimizationDashboard SHALL highlight the minimum objective value point on the chart and display the corresponding variable values.

---

### Requirement 9: Export Optimized Sections to Model

**User Story:** As a structural engineer, I want to apply the optimized section assignments back to the model, so that subsequent analysis and detailing use the optimized sections.

#### Acceptance Criteria

1. WHEN an optimization run completes, THE SensitivityOptimizationDashboard SHALL display an "Apply to Model" button that is enabled only when at least one section assignment differs from the current model.
2. WHEN the user clicks "Apply to Model", THE SensitivityOptimizationDashboard SHALL update the `sectionId` of each affected member in the model store to the optimized section designation.
3. WHEN sections are applied to the model, THE SensitivityOptimizationDashboard SHALL display a summary showing how many members were updated and the estimated weight change in kg.
4. THE SensitivityOptimizationDashboard SHALL allow the user to undo the section application by clicking a "Revert" button that restores the previous `sectionId` values within the same session.
5. WHEN sections are applied to the model, THE SensitivityOptimizationDashboard SHALL emit a model-change event so that the analysis results are marked as stale and the user is prompted to re-run analysis.

---

### Requirement 10: Detailing Overview Pass/Fail Dashboard

**User Story:** As a structural engineer, I want the detailing overview tab to show the actual design status of each member, so that I can immediately see which members pass or fail code checks without opening each tab individually.

#### Acceptance Criteria

1. WHEN the overview tab is active and analysis results are available, THE DetailingDesignPage SHALL display a table listing each member with columns for member ID, type (beam/column/brace), section, Utilization_Ratio, and pass/fail status.
2. THE DetailingDesignPage SHALL colour-code each row: green for Utilization_Ratio ≤ 0.85, amber for 0.85 < Utilization_Ratio ≤ 1.0, and red for Utilization_Ratio > 1.0.
3. WHEN the user clicks a member row in the overview table, THE DetailingDesignPage SHALL navigate to the appropriate design tab (beam, column, or steel) and pre-populate the design inputs with that member's section and force data.
4. THE DetailingDesignPage SHALL display a summary bar at the top of the overview showing total member count, pass count, fail count, and overall pass rate as a percentage.
5. WHEN no analysis results are available, THE DetailingDesignPage SHALL display a prompt directing the user to run structural analysis before viewing design status.

---

### Requirement 11: Batch Design Capability

**User Story:** As a structural engineer, I want to run code checks on all members of a given type in one action, so that I can quickly identify all failures without designing each member individually.

#### Acceptance Criteria

1. WHEN the user clicks "Auto-Design All Members" and analysis results are available, THE DetailingDesignPage SHALL run IS 456 or IS 800 code checks on every member in the model using the member's current section and the maximum force envelope from analysis results.
2. THE DetailingDesignPage SHALL complete batch design for up to 200 members within 5 seconds.
3. WHEN batch design completes, THE DetailingDesignPage SHALL update the overview pass/fail table with the results and display a toast notification showing the number of passing and failing members.
4. THE DetailingDesignPage SHALL store batch design results in component state so that the overview table remains populated when the user switches between tabs within the same session.
5. IF a member's section data is missing or incomplete, THEN THE DetailingDesignPage SHALL mark that member as "Skipped" in the batch results and continue processing remaining members.

---

### Requirement 12: Reinforcement Drawing Export

**User Story:** As a structural engineer, I want to export a reinforcement sketch for a designed RC member, so that I can share it with detailing draughtsmen or include it in a design report.

#### Acceptance Criteria

1. WHEN a beam or column design has been completed in the RC design panel, THE DetailingDesignPage SHALL display an "Export Drawing" button.
2. WHEN the user clicks "Export Drawing", THE DetailingDesignPage SHALL generate an SVG reinforcement sketch showing the cross-section with bar positions, bar diameters, cover dimensions, and stirrup/tie spacing.
3. THE reinforcement sketch SHALL include a title block with member ID, section dimensions, design code, and date.
4. THE DetailingDesignPage SHALL allow the user to download the reinforcement sketch as an SVG file.
5. WHERE the user's browser supports the File System Access API, THE DetailingDesignPage SHALL also offer a PDF download option by printing the SVG to a new browser window.

---

### Requirement 13: Design Report Generation

**User Story:** As a structural engineer, I want to generate a design report for all designed members, so that I can submit it as part of a structural design package.

#### Acceptance Criteria

1. WHEN at least one member has been designed (individually or via batch), THE DetailingDesignPage SHALL display a "Generate Report" button.
2. WHEN the user clicks "Generate Report", THE DetailingDesignPage SHALL compile an HTML report containing: project name, date, design code, a summary table of all designed members with their Utilization_Ratio and pass/fail status, and individual design calculation sheets for each designed member.
3. THE design report SHALL include the governing load combination, applied forces, section properties, and code check results for each member.
4. THE DetailingDesignPage SHALL open the generated report in a new browser window and trigger the browser print dialog so the user can save it as PDF.
5. THE design report SHALL be self-contained HTML with inline CSS so that it renders correctly when saved as a file without network access.

---

### Requirement 14: Round-Trip Layout Serialization

**User Story:** As a developer, I want the space planning project to serialize and deserialize without data loss, so that saved projects reload with identical room positions and MEP data.

#### Acceptance Criteria

1. THE SpacePlanningEngine SHALL serialize a `HousePlanProject` to a JSON string using `JSON.stringify`.
2. THE SpacePlanningEngine SHALL deserialize a JSON string back to a `HousePlanProject` using `JSON.parse`.
3. FOR ALL valid `HousePlanProject` objects, serializing then deserializing SHALL produce an object where every PlacedRoom has identical `x`, `y`, `width`, and `height` values to the original (round-trip property).
4. FOR ALL valid `HousePlanProject` objects, serializing then deserializing SHALL produce an object where every MEP fixture has identical `x`, `y`, and `roomId` values to the original.
5. IF a serialized project string is malformed or missing required fields, THEN THE SpacePlanningEngine SHALL return a descriptive error rather than a partially constructed object.
