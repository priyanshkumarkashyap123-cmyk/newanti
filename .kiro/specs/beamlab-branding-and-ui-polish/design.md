# BeamLab Branding and UI Polish â€” Bugfix Design

## Overview

Three distinct bugs share a common root: hardcoded brand strings that were never wired to the
single source of truth (`BrandingConstants.ts`) or the project store. The fix is deliberately
minimal â€” one constant change cascades through all PDF output (Part 1), a few lines in
`ReportsPage.tsx` wire the cover title and export handlers to the store (Part 2), and a targeted
audit pass ensures layout/navigation components consistently render the `<Logo>` component (Part 3).

## Glossary

- **Bug_Condition (C)**: The set of conditions that trigger incorrect branding output
- **Property (P)**: The desired correct behavior when a bug condition holds
- **Preservation**: Existing behaviors that must remain unchanged after the fix
- **BEAMLAB_COMPANY**: The constant object in `apps/web/src/constants/BrandingConstants.ts` that is the single source of truth for brand identity
- **projectInfo**: The `ProjectInfo` object in the model store (`useModelStore((s) => s.projectInfo)`) that holds the user's project name
- **handleExportPDF**: The export handler in `ReportsPage.tsx` that calls `generateProfessionalReport` / `generateDesignReport`
- **Logo**: The `<Logo>` component in `apps/web/src/components/branding/Logo.tsx` that renders the BeamLab Ultimate SVG assets

## Bug Details

### Bug Condition

Three independent bug conditions exist, each triggering a different branding defect.

**C1 â€” Wrong company name constant:**

```
FUNCTION isBugCondition_C1(constant)
  INPUT: constant of type typeof BEAMLAB_COMPANY
  OUTPUT: boolean

  RETURN constant.name === 'BeamLab'   // should be 'BeamLab Ultimate'
END FUNCTION
```

**C2 â€” Hardcoded cover title in ReportsPage:**

```
FUNCTION isBugCondition_C2(coverTitleSource)
  INPUT: coverTitleSource â€” the value rendered as the cover page h2 title
  OUTPUT: boolean

  RETURN coverTitleSource === 'BeamLab Project'   // literal string, not from store
END FUNCTION
```

**C3 â€” Hardcoded project name in PDF export handlers:**

```
FUNCTION isBugCondition_C3(exportArg)
  INPUT: exportArg â€” the name field passed to generateProfessionalReport / generateDesignReport
  OUTPUT: boolean

  RETURN exportArg === 'BeamLab Project'   // literal string, not from store
END FUNCTION
```

### Examples

- **C1**: Any PDF report cover page shows "BeamLab" in the top-left branding block instead of "BeamLab Ultimate"
- **C1**: The `@page` CSS running header reads "BeamLab â€” BL-01234" instead of "BeamLab Ultimate â€” BL-01234"
- **C2**: A user with a project named "Tower Block A" sees "BeamLab Project" as the cover page title
- **C3**: Exporting PDF for a project named "Bridge Deck" produces a PDF titled "BeamLab Project"
- **C3**: `handleExportIFC` and `handleExportExcel` also pass the hardcoded string (same root cause, same fix)

## Expected Behavior

### Preservation Requirements

**Unchanged Behaviors:**
- `BEAMLAB_COMPANY.generatedByLine` must continue to read `'Generated by BeamLab Ultimate â€” Professional Structural Engineering Software'` â€” it is already correct and must not be touched
- `BEAMLAB_COLORS`, `BEAMLAB_COLORS_RGB`, and `BEAMLAB_LOGO` constants must remain identical
- The `<Logo>` component's rendering logic, SVG asset paths, variant/size API, and accessibility attributes must remain unchanged
- PDF reports generated with a valid non-empty project name must continue to produce correctly structured A4 documents with all existing sections
- Pages that already correctly use the design system must not regress

**Scope:**
All behaviors that do NOT involve the three bug conditions above are completely unaffected. This includes:
- Mouse/keyboard interactions in the 3D workspace
- Analysis computation and result display
- Authentication and routing
- All other export formats (DXF, IFC, Excel) â€” though their project name argument is also fixed in Part 2, the export logic itself is unchanged

## Hypothesized Root Cause

1. **C1 â€” Stale constant value**: `BEAMLAB_COMPANY.name` was set to `'BeamLab'` when the product was rebranded to "BeamLab Ultimate". The `generatedByLine` field was updated but `name` was not. Single-character fix.

2. **C2 â€” Copy-paste placeholder never replaced**: The cover page `<h2>` in `ReportsPage.tsx` was written with a placeholder string `"BeamLab Project"` during initial scaffolding. The store selector for `projectInfo` was added to the component for other purposes but never connected to this render site.

3. **C3 â€” Same placeholder propagated to handlers**: `handleExportPDF`, `handleExportIFC`, and `handleExportExcel` were written alongside the cover page and share the same hardcoded string. The fix is to derive the project name from the store once and use it everywhere.

4. **Part 3 â€” Inconsistent Logo adoption**: Some layout components (`AppShell`, `ModernWorkspace`) already import and render `<Logo>` correctly. The audit is to confirm no key pages render a bare dark header without the logo, and to add it where missing.

## Correctness Properties

Property 1: Bug Condition C1 â€” Company Name Is 'BeamLab Ultimate'

_For any_ consumer of `BEAMLAB_COMPANY.name`, the fixed constant SHALL return the string
`'BeamLab Ultimate'`, causing all PDF cover pages, running headers, and footers that read
from this constant to display the correct product name.

**Validates: Requirements 2.1**

Property 2: Bug Condition C2 â€” Cover Title Reads From Store

_For any_ project store state where `projectInfo.name` is set to a non-empty string `N`,
the fixed `ReportsPage` cover page title SHALL render `N` (not the literal `'BeamLab Project'`).

**Validates: Requirements 2.2**

Property 3: Bug Condition C3 â€” PDF Export Uses Store Project Name

_For any_ project store state where `projectInfo.name` is set to a non-empty string `N`,
the fixed `handleExportPDF` SHALL pass `N` as the `name` field to both
`generateProfessionalReport` and `generateDesignReport` (not the literal `'BeamLab Project'`).

**Validates: Requirements 2.3**

Property 4: Preservation â€” Other Constants Unchanged

_For any_ consumer of `BEAMLAB_COMPANY`, `BEAMLAB_COLORS`, `BEAMLAB_COLORS_RGB`, or
`BEAMLAB_LOGO`, the fixed code SHALL return the same values as the original code for all
fields other than `BEAMLAB_COMPANY.name`.

**Validates: Requirements 3.1, 3.5**

Property 5: Preservation â€” PDF Structure Unchanged For Valid Inputs

_For any_ valid project name input (non-empty string), the fixed `generateDesignReport` /
`generateProfessionalReport` SHALL produce the same document structure (sections, tables,
page count) as the original, with only the project name field differing.

**Validates: Requirements 3.3**

## Fix Implementation

### Part 1 â€” `BrandingConstants.ts` (1 line)

**File**: `apps/web/src/constants/BrandingConstants.ts`

**Change**:
```ts
// Before
name: 'BeamLab',

// After
name: 'BeamLab Ultimate',
```

This single change cascades to every PDF cover page, running header, footer, and any other
component that reads `BEAMLAB_COMPANY.name`.

**Test mock update**: `apps/web/src/services/__tests__/PDFReportService.test.ts` mocks
`BEAMLAB_COMPANY` with `name: 'BeamLab'` â€” update to `name: 'BeamLab Ultimate'` to keep
the mock consistent with the real constant.

### Part 2 â€” `ReportsPage.tsx` (store wiring)

**File**: `apps/web/src/pages/ReportsPage.tsx`

**Specific Changes**:

1. **Add store selector** â€” derive project name once at the top of the component:
   ```ts
   const projectName = useModelStore((s) => s.projectInfo?.name) ?? 'Untitled Project';
   ```

2. **Cover page title** (~line 402) â€” replace literal with store value:
   ```tsx
   // Before
   <h2 ...>BeamLab Project</h2>

   // After
   <h2 ...>{projectName}</h2>
   ```

3. **Document control table** (~line 415) â€” replace literal with store value:
   ```tsx
   // Before
   <td ...>BeamLab Project</td>

   // After
   <td ...>{projectName}</td>
   ```

4. **`handleExportPDF`** â€” replace both `name: 'BeamLab Project'` occurrences:
   ```ts
   { name: projectName, engineer: userName, ... }
   ```

5. **`handleExportIFC` and `handleExportExcel`** â€” same substitution:
   ```ts
   generateIFC({ name: projectName, author: userName }, ...)
   exportProjectData(projectName, ...)
   ```

6. **Executive summary paragraph** (~line 551) â€” replace inline literal:
   ```tsx
   // Before
   ...for the BeamLab Project model...

   // After
   ...for the {projectName} model...
   ```

## Testing Strategy

### Validation Approach

Two-phase: first run exploratory tests on unfixed code to confirm the bug conditions and
root cause analysis, then verify the fix and run preservation checks.

### Exploratory Bug Condition Checking

**Goal**: Surface counterexamples on unfixed code to confirm root cause before implementing the fix.

**Test Plan**: Import `BEAMLAB_COMPANY` directly and assert the name; render `ReportsPage` with
a known store state and assert the cover title; spy on `generateProfessionalReport` and assert
the `name` argument. Run on unfixed code â€” all three should fail.

**Test Cases**:
1. **C1 Exploration**: Assert `BEAMLAB_COMPANY.name === 'BeamLab Ultimate'` â€” will fail on unfixed code (returns `'BeamLab'`)
2. **C2 Exploration**: Render `ReportsPage` with `projectInfo.name = 'Tower A'`, assert cover `<h2>` text equals `'Tower A'` â€” will fail (shows `'BeamLab Project'`)
3. **C3 Exploration**: Spy on `generateProfessionalReport`, trigger `handleExportPDF`, assert `spy.calls[0][0].name === 'Tower A'` â€” will fail (passes `'BeamLab Project'`)

**Expected Counterexamples**:
- `BEAMLAB_COMPANY.name` returns `'BeamLab'` instead of `'BeamLab Ultimate'`
- Cover title renders literal `'BeamLab Project'` regardless of store state
- Export handler passes `'BeamLab Project'` regardless of store state

### Fix Checking

**Goal**: Verify that for all inputs where each bug condition holds, the fixed code produces the expected behavior.

**Pseudocode:**
```
FOR ALL constant WHERE isBugCondition_C1(constant) DO
  ASSERT BEAMLAB_COMPANY_fixed.name === 'BeamLab Ultimate'
END FOR

FOR ALL storeState WHERE isBugCondition_C2(coverTitleSource) DO
  result := render(ReportsPage_fixed, storeState)
  ASSERT coverTitle(result) === storeState.projectInfo.name
END FOR

FOR ALL storeState WHERE isBugCondition_C3(exportArg) DO
  result := captureExportArg(handleExportPDF_fixed, storeState)
  ASSERT result.name === storeState.projectInfo.name
END FOR
```

### Preservation Checking

**Goal**: Verify that for all inputs where the bug conditions do NOT hold, the fixed code produces the same result as the original.

**Pseudocode:**
```
FOR ALL constant WHERE NOT isBugCondition_C1(constant) DO
  // All other BEAMLAB_COMPANY fields
  ASSERT BEAMLAB_COMPANY_fixed[field] === BEAMLAB_COMPANY_original[field]
END FOR

FOR ALL input WHERE NOT isBugCondition_C3(input) DO
  // Non-PDF export paths (DXF, IFC structure, Excel structure)
  ASSERT exportHandler_original(input) === exportHandler_fixed(input)
END FOR
```

**Testing Approach**: Property-based testing is recommended for preservation of the constants
because it can generate many field-access patterns and verify none are broken. For the PDF
structure preservation, a targeted integration test with a known project name is sufficient.

### Unit Tests

- Assert `BEAMLAB_COMPANY.name === 'BeamLab Ultimate'` (C1 fix check)
- Assert all other `BEAMLAB_COMPANY` fields are unchanged (C1 preservation)
- Render `ReportsPage` with various `projectInfo.name` values; assert cover title matches (C2 fix check)
- Render `ReportsPage` with `projectInfo.name = undefined`; assert cover title falls back to `'Untitled Project'` (C2 edge case)
- Spy on `generateProfessionalReport`; trigger `handleExportPDF`; assert `name` arg matches store (C3 fix check)

### Property-Based Tests

- **Property 1**: For any non-empty string `name`, `BEAMLAB_COMPANY.name` always equals `'BeamLab Ultimate'` (constant, not input-dependent â€” verifies the fix is not accidentally dynamic)
- **Property 2**: For any `projectInfo.name` string drawn from an arbitrary generator, the rendered cover title equals that string
- **Property 3**: For any `projectInfo.name` string, `handleExportPDF` passes that exact string as the `name` argument to the PDF generator
- **Property 4**: For any field key in `BEAMLAB_COMPANY` other than `name`, the value is identical before and after the fix

### Integration Tests

- Generate a PDF report end-to-end with `projectInfo.name = 'Integration Test Project'`; assert the PDF text contains `'Integration Test Project'` and `'BeamLab Ultimate'`
- Render `ReportsPage` inside a full store provider; verify cover title, document control table, and running header all show the store project name
- Verify `AppShell` and `ModernWorkspace` render `<Logo>` in their headers without regression after Part 3 audit changes
